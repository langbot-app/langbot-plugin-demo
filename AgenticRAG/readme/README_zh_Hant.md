# AgenticRAG

AgenticRAG 將為目前管道配置的知識庫公開為 LLM 工具，因此代理可以檢查可用的 KB 並根據需要檢索相關區塊。

## 它的作用

- 提供單一工具“query_knowledge”
- 支援兩種操作：
  -`list`：列出目前管道可用的知識庫
  -`query`：從一個或多個選定的知識庫中檢索相關文檔
- 以 JSON 形式傳回檢索結果，以便代理可以繼續對它們進行推理

## 整體設計

AgenticRAG 故意不是一個新的 RAG 後端。它是一個控制層插件，可以改變**誰決定何時應該進行檢索**。

AgenticRAG 不是讓運行程式總是在模型回應之前自動注入知識，而是將檢索轉移到代理循環：

- 模型先決定是否需要檢索
- 該模型可以檢查可用的知識庫
- 模型可以選擇一個 KB 或平行查詢多個
- 檢索僅在模型明確要求時發生

這種設計的存在是為了解決一個特定的問題：天真的永遠在線檢索很簡單，但它也引入了噪音、浪費的上下文和許多回合的不相關塊。

＃＃ 設計

這個插件故意做得很薄。它沒有實現新的 RAG 後端。相反，它包裝了 LangBot 的內建查詢範圍知識檢索 API：

-`list_pipeline_knowledge_bases()`用於枚舉目前查詢可見的知識庫
-`retrieve_knowledge()`用於從一個或多個選定的 KB 檢索前 k 個條目

當呼叫該工具時，「query_id」由運行時注入，然後儲存在「QueryBasedAPIProxy」中。因此，工具程式碼只需要傳遞「kb_id」或「kb_ids」、「query_text」和「top_k」等業務參數。

儘管底層運行時可以支援元資料過濾器，但此外掛程式不會將原始過濾器暴露給目前代理工具流程中的代理程式。不同的知識引擎和向量後端可能使用不同的元資料欄位、值格式和過濾語義，而代理目前沒有這些欄位的可靠模式來源。

在生態系統有更統一的方式來描述每個知識庫的可過濾字段和運算符之後，未來的版本可能會公開元資料過濾。

## 它是如何工作的

AgenticRAG 請求有四個主要階段：

### 1. 停用簡單檢索

在「PromptPreProcessing」期間，插件會檢查活動的LLM是否支援工具呼叫。

- 如果支援工具調用，它會清除跑步者的“_knowledge_base_uuids”，因此會跳過正常的天真的預檢索步驟
- 如果不支援工具調用，它會保持啟用 naive RAG 作為後備，以便 KB 檢索不會完全消失

### 2.將檢索策略注入系統提示字元中

同時，AgenticRAG 注入一個額外的系統提示，告訴模型：

- 配置的知識庫是範圍內事實的主要事實來源
- 沒有自動檢索回退
- 對於事實、政策、程序、產品和特定領域的問題，它應該更喜歡“query_knowledge”

這很重要，因為單獨的工具描述通常不足以可靠地改變模型行為。

### 3.讓模型檢查和查詢知識庫

然後，代理商可以分兩步驟使用「query_knowledge」：

-`action="list"`查看哪些 KB 可用
-`action="query"`並行搜尋 1 KB 或數 KB

對於單 KB 檢索，首選參數是「kb_id」。
對於多 KB 檢索，請使用「kb_ids」。

### 4. 以結構化 JSON 傳回檢索結果

工具合併結果，以「knowledge_base_id」對其進行註釋，並傳回 JSON，以便模型可以繼續推理、工具使用或最終答案。

## 檢索行為

啟用 AgenticRAG 後，它會停用運行器對目前管道的自動幼稚 RAG 預處理。

- 在模型回答之前不再自動執行檢索
- 是否查詢知識庫現在是透過「query_knowledge」深思熟慮的模型決定
- 如果模型不呼叫該工具，則不會將任何知識庫內容注入到上下文中

有一個重要的例外：

- 如果活動 LLM 不支援工具調用，AgenticRAG 會保持啟用 naive RAG，而不是停用它

這減少了無條件檢索噪音，但也意味著提示問題。因此，目前的實作使用**兩者**：

- 關於「query_knowledge」的工具提示
- 在「PromptPreProcessing」期間注入的系統提示

它們共同使模型偏向事實、政策、程序、產品和其他特定領域問題的檢索。

## 為什麼要這樣設計

該插件針對特定權衡進行了最佳化：

- 保留LangBot現有的知識庫與擷取基礎設施
- 刪除不必要的始終在線檢索
- 讓模型做出明確的檢索決策
- 仍然將檢索行為限制在目前管道中

與樸素的 RAG 相比，此設計為您提供：

- 在不需要知識庫存取的回合中減少不相關的上下文
- 更能控制查詢哪個知識庫
- 迭代檢索、重新查詢和多 KB 推理的空間

缺點也是真實的：如果模型從不呼叫該工具，則不會出現知識庫內容。這就是為什麼外掛程式明確添加面向檢索的提示，而不是假設模型會足夠頻繁地自然選擇檢索。

這也是為什麼插件現在在禁用 naive RAG 之前檢測工具呼叫功能的原因。如果沒有這個防護，在不支援工具的模型上啟用 AgenticRAG 會意外地完全破壞 KB 檢索。

## 安全邊界

此工具的範圍僅限於目前管道。

- LangBot運作時也會在執行檢索之前驗證要求的「kb_id」是否屬於目前管道

這意味著僅提示注入不應允許代理查詢管道配置之外的任意 KB。

## 如何使用

1.安裝並啟用插件。
2. 在目前管道的本機代理設定中配置一個或多個知識庫。
3. 讓代理呼叫`query_knowledge`：
   - 從`action="list"`開始檢查可用的 KB
   - 然後使用`kb_id`呼叫`action="query"`來呼叫 1 KB，或呼叫`kb_ids`來並行查詢多個 KB
   - 為合併結果計數提供“query_text”和可選的“top_k”

＃＃ 參數

對於`action="query"`，該工具目前接受：

-`kb_id`：單KB檢索的目標知識庫UUID；查詢剛好 1 KB 時首選
-`kb_ids`：用於平行多 KB 檢索的可選目標知識庫 UUID 陣列；僅在查詢多個 KB 時使用
-`query_text`：檢索查詢文本
-`top_k`：可選正整數，預設`5`，套用於合併結果集

如果一個 KB 查詢失敗而其他查詢成功，工具會傳回一個包含「results」和「failed_kbs」的 JSON 對象，以便代理可以繼續處理部分結果。

## 典型流程

1. 代理程式列出可用的 KB。
2. 代理人根據名稱和描述選擇一個 KB 或一組 KB。
3. 代理提交集中檢索查詢。
4. 代理程式使用傳回的區塊來應答或繼續工具使用。

## 提示意圖

提示層旨在向模型傳達兩件事：

- 這些知識庫是範圍內資訊的權威來源
- 啟用 AgenticRAG 後，不存在後備自動檢索

如果沒有這種指導，法學碩士可能會過度信任其預先訓練的知識而導致檢索使用不足。因此，目前的實作在系統提示層和工具提示層都強化了相同的策略。

## 日誌記錄

該插件現在會在工具執行期間發出日誌，以便您可以觀察 LLM 在實踐中如何使用 AgenticRAG。

您將看到以下日誌：

- 工具呼叫開始，包括`query_id`、`action`和參數鍵
- KB 清單開始/結束以及可見的 KB 數量
- 檢索開始，包括選定的 KB、「top_k」和縮短的「query_text」預覽
- 按KB檢索開始/成功/失敗
- 最終檢索摘要，包括合併結果數、失敗KB數和傳回結果數

典型的日誌訊息如下所示：

```text
[AgenticRAG] tool call started: query_id=123 action=query params_keys=['action', 'kb_id', 'query_text', 'top_k']
[AgenticRAG] retrieval requested: query_id=123 kb_ids=['kb-1'] kb_count=1 top_k=5 query='what is the refund policy'
[AgenticRAG] querying knowledge base: query_id=123 kb_id=kb-1 top_k=5
[AgenticRAG] knowledge base query succeeded: query_id=123 kb_id=kb-1 result_count=4
[AgenticRAG] retrieval completed: query_id=123 merged_results=4 failed_kbs=0
```
